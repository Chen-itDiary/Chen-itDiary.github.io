JVM

JVM、JRE、JDK关系

JVM屏蔽了底层操作系统的差异，实现了“**一次编译，到处运行**”

程序计数器

定义

  • 作用，记住下一条jvm指令（字节码文件）的执行地址
  • 特点
    • 是线程私有的（每个线程都有自己的程序计数器）
    • 不会存在内存泄漏的问题

虚拟机栈

定义

Java Virtual Machine Stacks（java虚拟机栈）

  • 每个线程运行所需要的内存，被称之为虚拟机栈
  • 每个栈可由多个栈帧（Frame）组成，对应着每个方法调用时所占用的内存
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的方法

栈内存溢出

  • 栈帧过多导致栈内存溢出
  • 栈帧过大导致栈内存溢出

线程运行诊断

案例1：cpu占用过多

定位

  • 用top命令定位哪儿个进程对cpu的占用过高
  • ps H -eo pid,tid,%cpu | grep 进程id（用ps命令进一步定位哪儿个线程引起的cpu占用过高）
  • jstack 进程id
    • 可以根据线程id找到有问题的线程，进一步定位到问题代码的源码行号

案例2：程序运行很长时间没有结果

本地方法栈

本地方法非java实现，底层由c/c++实现

调用本地方法有本地方法栈来管理其内存

堆

定义

Heap堆

  • 通过new关键字，创建对象都会用到堆内存

特点

  • 线程共享，堆中对象考虑线程安全
  • 拥有垃圾回收机制

堆内存溢出

堆内存诊断

1.jps

  • 查看当前系统中的java进程

2.jmap

  • 查看堆内存占用 -heap 进程id

3.jconsole

  • 图形界面，可以连续监测

案例:

jvisualvm工具–可快照查看有多少类、内存的分布……

组成

1.6版本方法区属于堆内存的一部分；1.8版本及以后使用本地操作系统内存

方法区内存溢出

1.8以前会导致永久代内存溢出

1.8以后会导致元空间内存溢出

运行时常量池

  • 常量池，就是一张表，虚拟机指令根据这张常量表去找到要执行的类名、方法名、参数类型、字面量等信息
  • 运行时常量池，常量池是*.class文件中的，当该类被加载，它的常量池就会被放入运行时常量池，并把里面的符号地址变为真实地址

StringTable

懒惰加载，遇到加载该字符串时才生成字符串对象并放入StringTable【结构：hashtable，无法扩容】

特性：

  • 常量池中的字符串仅是符号，在第一次用到才变为对象
  • 利用串池的机制，避免重复创建字符串对象
  • 变量的字符串拼接原理是StringBuilder（1.8)—拼接后的字符串仅仅存在堆中
  • 字符串常量拼接的原理是编译器优化–直接寻找StringTable中相同的字符串
  • 可以使用intern方法，主动将串池中还没有的字符串对象放入串池

StringTable垃圾回收

StringTable性能调优

  • 调整桶个数 -XX:StringTableSize=桶个数
  • 考虑将字符串对象是否入池

直接内存

定义

Direct Memory

  • 常见于NIO操作时，用于数据缓冲区
  • 分配回收成本较高，但读写性能高
  • 不受JVM内存回收管理

回收方式（释放原理）

通过Clean对象（虚引用）将DirectByteBuffer和回收内存任务关联，当DirectByteBuffer对象被垃圾回收后，直接触发Clean对象的clean()方法，利用Unsafe清除掉直接内存.

垃圾回收

如何判断一个对象可以被回收

引用计数法

每被引用一次，计数+1，计数为0即表示可回收

可达性分析算法

  • Java虚拟机种的垃圾回收器通过可达性分析来探索所有存活的对象
  • 扫描堆中的对象，看是否能够沿着GC Root对象为起点的引用链找到该对象，找不到，表示可以回收
  • 哪儿些对象可以作为GC Root？

四种引用

  1. 强引用
  2. 软引用
  3. 弱引用
  4. 虚引用
  5. 终结器引用

对于A1被两个GC Root对象直接连接的，表示强引用

如果只有一个软引用连接A2，没有强引用，当垃圾回收后发现空间仍旧不够，则会继续回收A2对象、

如果只有一个弱引用连接A3，没有强引用，当垃圾回收时就会直接回收该对象

软弱引用可配合引用队列使用：当软弱引用不再被引用，可放入引用队列中，通过引用队列查询软弱引用位置并可用垃圾回收清除

虚引用、终结器引用必须配合引用队列使用

当虚引用引用的对象被垃圾回收时，虚引用就会放入引用队列，最后通过某线程定时查看引用队列并执行相对应Cleaner的方法清除掉直接内存

当A4对象重写了finallize()方法同时不再被强引用引用该对象时，虚拟机会创建一个终结器引用，引用该对象。当A4对象发生垃圾回收时，终结器引用被放入引用队列，最终仍旧由某一线程查询引用队列并发现终结器引用后，再根据终结器引用找到A4 对象并调用该方法的finallize()方法，在下一次垃圾回收时就可以回收掉该对象.（）

补充

垃圾回收算法

标记清除法

优点：速度快

缺点：产生内存碎片—空间不连续

标记整理法

优点：不会产生内存碎片

缺点：清除速度慢，涉及对象引用地址的移动

复制法

优点：不产生内存碎片

缺点：使用双倍的内存空间

垃圾回收器

  1. 串行

    • 单线程（垃圾回收，关闭其他线程，开放一个线程专门进行垃圾回收）
    • 堆内存较小，适合个人电脑

  2. 吞吐量优先

    • 多线程
    • 堆内存较大，适合多核CPU
    • 让单位时间内，STW 的时间最短

  3. 响应时间优先

    • 多线程

    • 堆内存较大，适合多核CPU

    • 尽可能让单次 STW 的时间最短

串行

Serial回收工作在新生代，采用复制算法

SerialOld回收工作在老年代，采用标记+整理算法

吞吐量优先

响应时间优先

G1

Young Collection

  • 会STW

Young Collection + CM

  • 在Young GC时会进行GC Root 的初始标记
  • 在老年代占用堆空间比例达到阈值时，进行并发标记（不会STW），由下面参数决定老年代占用堆空间阈值

Mixed Collection

会对E、S、O进行全面垃圾回收

  • 最终标记会STW
  • 拷贝存活会STW

Full GC

CMS在并发清除垃圾失败时才会触发Full GC

而G1则是并发垃圾回收速度跟不上产生垃圾的速度时，会触发Full GC

Young Collection 跨代引用

  • 卡表与Remember Set（存储谁引用了E中对象）
